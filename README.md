# IEEE-CIS Fraud Detection

## პროექტის მიმოხილვა

ამ Kaggle-ის კონკურსის მიზანია საბანკო ტრანზაქციებში თაღლითობის (fraud) გამოვლენა. მონაცემები წარმოდგენილია ორი ცხრილით: Transaction და Identity, რომლებიც TransactionID-ის მიხედვით ერთმანეთთან არის დაკავშირებული. ეს არის კლასიფიკაციის ამოცანა, რომელიც ფასდება ROC-AUC მეტრიკით.

## მონაცემების დამუშავება

**გაერთიანება**
სატრანზაქციო და საიდენტიფიკაციო ცხრილები გაერთიანდა LEFT JOIN-ით TransactionID-ის მიხედვით, რადგან მოცემული competition-ის დატასეტი ორი ცხრილით იყო მოცემული, რომელთა feature-ებსაც ერთი და იგივე საერთო TransactionID ჰქონდათ, ამით ორი ცხრილი გარდავქმენით ერთ დიდ ტაბულარულ მონაცემად, რომელზეც შესაძლებელი გახდა Feature Engineering და მოდელის ტრენინგი.

**მეხსიერების ოპტიმიზაცია**
დატასეტი შეიცავს ~590,000 სტრიქონს და 400+ სვეტს. მეხსიერების შესამცირებლად გამოყენებული იქნა `reduce_mem_usage` ფუნქცია, რომელსაც სვეტები ოპტიმალურ ცვლადის ტიპებზე გადაყავს. ეს საჭირო გახდა მაღალი RAM-ის მოხმარების გამო.

**გამოტოვებული მნიშვნელობები**
- სვეტები, რომელთა NA-ების წილი 80%-ს აჭარბებდა, ამოვაგდე
- კატეგორიული სვეტები: NA შევავსე `"Unknown"` ტექსტით, რათა WOE ენკოდერს ცალკე კატეგორიად აღექვა
- რიცხვითი სვეტები: NA შევავსე მედიანით `SimpleImputer`-ის გამოყენებით

## Feature Engineering

**WOE ენკოდინგი**
კატეგორიული ცვლადების დასამუშავებლად გამოყენებული იქნა Weight of Evidence (WOE) ენკოდინგი. WOE განსაკუთრებით ეფექტურია თაღლითობის გამოვლენის ამოცანებში, რადგან პირდაპირ ასახავს კატეგორიული ცვლადის კავშირს საბოლოო (გამოსაცნობ) ცვლადთან.

**OneHotEncoding**
დაბალი უნიკალური მნიშვნელობების რაოდენობის მქონე კატეგორიული სვეტებისთვის გამოყენებული იქნა OneHotEncoding, რათა სვეტების საერთო რაოდენობა არ გაზრდილიყო ზედმეტად. უნიკალური მნიშვნელობების ზღვარი = 3.

**StandardScaler**
ყველა რიცხვითი სვეტისთვის გამოვიყენე StandardScaler, განსაკუთრებით Logistic Regression მოდელისთვის, რომელიც მასშტაბის მიმართ მგრძნობიარეა.

## Feature Selection

**Information Value (IV)**
IV ფილტრი გამოყენებული იქნა კატეგორიული სვეტების გასაფილტრად. IV ზღვრად ავიღე 0.05/0.1, რის ქვემოთაც სვეტები "უსარგებლოდ" ითვლება პროგნოზირებისთვის. ზღვარი იცვლებოდა ექსპერიმენტების მიხედვით, IV ფილტრის ეფექტურობის შესამოწმებლად.

**კორელაციის ფილტრი**
კორელაციის ფილტრის ზღვრად ავიღე 0.85/0.9. ამაზე მაღალი კორელაციის შემთხვევაში ერთ-ერთ სვეტს ვაგდებდი, რათა თავი ამერიდებინა multicollinearity-სთვის და overfitting-სთვის. ზღვარი იცვლებოდა ექსპერიმენტების მიხედვით, კორელაციის ფილტრის შესამოწმებლად

**RFE (Recursive Feature Elimination)**
RFE თითოეული მოდელისთვის გამოვიყენე, მხოლოდ ერთხელ ეფექტურობის შესამოწმებლად, სხვა არცერთ ექსპერიმენტში არ გამომიყენებია, რადგან RFE-ს feature importance-ების დასადგენად ძალიან დიდი დრო სჭირდებოდა.

## კლასთა დისბალანსი

დატასეტი მაღალი დისბალანსით გამოირჩევა — fraud ტრანზაქციები მხოლოდ ~3.5%-ს შეადგენს. ამის მოსაგვარებლად გამოვიყენე -->

- **Undersampling**: გამოვიყენე majority undersampling, რათა გამეთანაბრებინა დატასეტში isFraud = 0 და isFraud = 1 დატა ფოინთების რაოდენობა.

## ტრენინგი

არ გამომიყენებია GLM, Linear Regression და DecisionTree, რადგან კლასიფიკაციის:

- **Linear Regression**: ჩავთვალე, რომ კლასიფიკაციის ამოცანისთვის ტყუილი დროის ხარჯვა იქნებოდა, მითუმეტეს სხვა მოდელების გაცნობის ფონზე

- **GLM**: მაინც Logistic Regression-მდე მივიდოდა

- **DecisionTree**: XGBoost და Random Forest-ის ფონზე ამ მოდელის შემოწმება ისევ და ისევ ტყუილი დროის ხარჯვა გამოვიდოდა


საბოლოოდ, სამი მოდელი შედარდა ერთმანეთთან იდენტური preprocessing pipeline-ით, რომელიც სხვადასხვა ექსპეტიმენტებში იცვლებოდა, ფილტრების პარამეტრების შეცვლით:


**XGBoost**
```
n_estimators: [100, 300, 500]
max_depth: [3, 5, 7]
learning_rate: [0.01, 0.1, 0.3]
scale_pos_weight: ~28
```

**Random Forest**
```
n_estimators: [100, 300, 500]
max_depth: [5, 10, 20]
min_samples_split: [2, 5, 10]
max_features: ['sqrt', 'log2']
```

**Logistic Regression**
```
C: [0.01, 0.1, 1, 10]
penalty: ['l1', 'l2']
solver: ['saga']
max_iter: [500, 1000]
```

ყველა მოდელისთვის გამოყენებული იქნა `GridSearchCV` 3-fold cross-validation-ით, რამაც ხელი შემიწყო ერთდროულად შემემოწმებინა ბევრი სხვადასხვა პარამეტრების მქონე მოდელი. შეფასების მეტრიკად გამოვიყენე ROC-AUC, რადგან კეგლზეც ამით ფასდებოდა საბოლოო შედეგი.
